# node-webview 常见问题及解决方案

## 1. WKWebView debugger 暂停导致 evaluateJavaScript 失效

### 现象

WebView 端开启 debugger 暂停后，Host 端通过 `evaluateScript` 发送的脚本无法在 WebView 中执行，消息丢失。

### 根因

wry 0.55.1 的 `evaluate_script`（fire-and-forget 路径）内部调用 WKWebView 的 `evaluateJavaScript:completionHandler:` 时，传入 **NULL** 作为 `completionHandler`：

```rust
// wry 内部实现（wkwebview/mod.rs）
#[cfg(not(feature = "tracing"))]
let handler: Option<block2::RcBlock<dyn Fn(*mut AnyObject, *mut NSError)>> = None;
self.webview.evaluateJavaScript_completionHandler(&NSString::from_str(js), handler.as_deref());
```

当 JS 引擎被 `debugger` 暂停时，WKWebView 对 NULL handler 的 `evaluateJavaScript` 调用处理不可靠，可能直接丢弃脚本而非排队等待恢复。

### 解决方案

在 `src/rs/window.rs` 的 `evaluate_script` 方法中，改用 `evaluate_script_with_callback` 并传入非 NULL 的 dummy callback：

```rust
pub fn evaluate_script(&self, js: &str) -> wry::Result<()> {
    // 使用非 NULL 的 dummy callback，确保 WKWebView 在 debugger 暂停期间
    // 也能可靠地排队脚本，待恢复后依次执行
    self.webview.evaluate_script_with_callback(js, |_| {})
}
```

### 影响文件

- `src/rs/window.rs` — `evaluate_script` 方法

## 2. Host handler 中 messages + return 导致 request Promise 不 resolve

### 现象

Host 端 request handler 中先调用 `win.rpc.messages.eventName()`（fire-and-forget 消息），再 `return` 响应值。如果 WebView 端的消息 handler 包含 `debugger`，则 WebView 端发起的 `rpc.requests.xxx()` Promise 永远不 resolve。

### 触发场景

```typescript
// Host 端
const rpc = defineRPC<RPC>({
  requests: {
    echo: async data => {
      win.rpc?.messages.update({ message: 'host-reply' })  // 触发 _handleSend
      return { received: data.msg }                         // 触发 _resolve
    }
  }
})

// WebView 端
rpc.on('update', async data => {
  debugger  // 暂停 JS 引擎
})

await rpc.requests.echo({ msg: 'hello' })  // 永远不返回
```

### 根因

Host handler 中 `messages.update()` 和 `return` 产生两次独立的 `evaluateScript` 调用：

1. **`_handleSend`** — 触发 WebView 消息 handler（含 `debugger`）
2. **`_resolve`** — 解析 WebView 端 request Promise

两者作为独立的 `evaluateJavaScript` 调用发往 WebKit。执行流程：

```
evaluateJS("_handleSend(...)")  →  handler 命中 debugger → JS 暂停
evaluateJS("_resolve(...)")     →  排队等待，无法执行
```

`_handleSend` 暂停 JS 引擎后，`_resolve` 虽然被排队但无法执行，导致 WebView 端的 request Promise 永远挂起。

### 解决方案

在 `src/ts/window.ts` 的 `sendToWebview` 方法中，用 `setTimeout(fn, 0)` 包裹 `_handleSend` 调用：

```typescript
sendToWebview(event: string, data?: any): void {
    const payload = JSON.stringify(data === undefined ? null : data)
    // setTimeout 使 _handleSend 在独立 macrotask 中执行，
    // evaluateScript 瞬间完成（仅注册定时器），_resolve 可立即执行
    this.evaluateScript(
      `setTimeout(function(){ window.__nodeWebview && window.__nodeWebview._handleSend(${JSON.stringify(event)}, ${payload}) }, 0)`
    )
}
```

执行流程变为：

```
evaluateJS("setTimeout(_handleSend, 0)")  →  瞬间完成（仅注册定时器）
evaluateJS("_resolve(...)")               →  正常执行，Promise resolve
[当前 tick 结束]
setTimeout 回调触发                        →  _handleSend 执行 → debugger 暂停
```

`_resolve` 和 `_handleSend` 在不同的 JS macrotask 中执行，`debugger` 只影响自身所在的 task，不会阻塞 `_resolve`。

### 影响文件

- `src/ts/window.ts` — `RPC_BRIDGE_SCRIPT` 中的 `_handleSend` 方法

## 3. rpc.messages.eventName() 从 Promise 改为同步调用

### 背景

`rpc.messages` 上的方法为 fire-and-forget 模式（仅发送消息，不等待回复），但原实现返回 `Promise<void>`，语义不正确且引入不必要的异步开销。

### 修改内容

**类型层面**（`src/ts/types.ts` + `src/ts/client.ts`）：

`RPCPromise` 类型区分 `messages` 和 `requests`：
- `messages` 分支：`(...args: A) => void`（同步）
- `requests` 分支：`(...args: A) => Promise<Awaited<R>>`（保持异步）

**运行时层面**（`src/ts/window.ts`）：

- Host 端 `sendToWebview` 返回 `void`（非 `Promise<void>`）
- 桥接脚本 `rpcMessages` proxy 不再 `return Promise.resolve()`

### 影响文件

- `src/ts/types.ts` — `RPCPromise` 类型、`HostRPCInstance`/`WebviewRPCInstance` 接口
- `src/ts/client.ts` — 全局声明、本地 `RPCPromise` 定义
- `src/ts/window.ts` — `sendToWebview` 方法、桥接脚本 `rpcMessages` proxy

## 4. TypeScript 类型文件被外部进程反复还原

### 现象

通过 `Edit` 工具对 `src/ts/window.ts` 的修改会被外部进程（可能是 IDE linter 或 file watcher）在短时间内还原。

### 解决方案

对于被反复还原的文件，使用以下策略：

1. **`Write` 工具** — 原子写入完整文件内容，减少还原窗口
2. **`sed` 命令** — 通过 Bash 直接操作文件，绕过部分 watcher
3. **Python 脚本** — 通过 Python `open/write` 原子替换文件内容

对于桥接脚本字符串内的修改，优先考虑在 TypeScript 代码层面实现同等效果（如 `sendToWebview` 中用 `setTimeout` 包裹），避免直接修改模板字符串。
